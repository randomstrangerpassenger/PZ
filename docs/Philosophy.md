1. Pulse - 모드로더 겸 플랫폼의 역할 [기반적 기능]

Pulse의 유일한 정체성은 "자유도". 완전한 샌드박스형 모딩 플랫폼이 목표임

아무 목표도 없는, 다만 무너지지 않고 탄탄한 기반만 제공하는 '휑한 들판'같은 플랫폼

오로지 기반적 기능만 가져야하고 이를 API 등의 형태로 타 모드에 제공해야함

모드적 기능은 포함하면 절대 안됨

-> 버전 업데이트에 강한 내성을 가지게 됨

추가할 만한 편의성 기능들은 모두 Cortex로 이전.

목표는 Fabric형 얇은 플랫폼

결정/정책은 절대 하지 않음.

퍼스트 파티 모드라 잘 돌아가는게 아닌 플랫폼이 좋아서 퍼스트 파티 모드가 잘 돌아가도록 만들어야함.

목표는 기존의 모딩 생태계(Lua 기반 모드)와 새로운 모딩 생태계(Mixin 기반 모드)를 융합하여 모더들에겐 더 큰 모딩 자유도를, 일반 유저들에겐 더 큰 모드 선택지를 제공하는것.

5. Iris - [100% Lua를 사용하는 위키피디아형 모드]

"질문은 다르되, 대답의 태도는 같다"

기본 바닐라 툴팁 위에 Alt 키를 눌렀을 때만 Iris 툴팁이 표시되어 용도와 더 자세한 설명이 나오도록 해야함. 길이는 최대 4줄.

아이템을 오른 클릭을 했을때 "Iris 메뉴에서 더보기" 버튼을 클릭했을때 Iris 모드 메뉴(위키)가 나타나 더욱 세부적인 정보를 제공(실용적인 정보만)

Iris 메뉴는 대분류 - 소분류 - 아이템 목록 - 아이템 설명 순으로 구성될 예정이고 각 칸의 크기는 아이템 설명 > 대분류 > 소분류 = 아이템 목록 순으로 이루어질 예정

위키피디아 모드인 이상 게임플레이에 실질적인 도움이 되는 정보를 담아야 함.

대신 해석, 권장, 비교는 엄격히 금함.

Iris 메뉴에 담기는 정보들은 전부 툴팁보다는 상세해야하며 툴팁의 확장선이라는 느낌보단 위키라는 느낌의 정보가 들어있어야함.

Iris 시스템은 총 3단계로 구성됨

그리고 그 각 단계 또한 여러개의 층위로 구성됨

1단계 증거 시스템은 2개의 동일 2트랙으로 구성됨

1-1. 레시피 기반 증거 시스템 1-2. 우클릭 행동 기반 증거 시스템

이 두 계층은 서로 독립적이며, 둘 사이엔 우열이 없는 증거 시스템 하위의 동급 시스템이어야 함.

2단계 아이템 분류 시스템

2-1. 대분류 2-2. 소분류 2-3. 아이템 목록 2-4. 아이템 설명란

3단계 아이템 설명란

3-1. 아이템의 기초 설명 3-2. 주 소분류 설명 3-3. 아이템의 개별 설명 3-4. 아이템의 관련 레시피나 관련 우클릭 행동 등의 아이템의 상호작용 3-5. 아이템의 내부 정보 (PZ 내부 분류 정보, Iris 내부 분류 정보 등)

아이템 설명란의 각 계층은 선행 계층의 필터 결과물이 아니라, 각자 고유한 정보 유형을 담당하는 독립 층임. 다만 실용성 기준상 1계층과 2계층만으로 용도 설명이 충분한 경우 3계층과 4계층은 침묵할 수 있음. 5계층은 내부 정보층이므로 항상 존재함.
단, 고유 정보 유형이라고 해서 굳이 타 계층을 배제하여 서술할 필요는 없음.

추후 모드 시장 확장 시스템 역시 구현할 예정

전략은 바닐라와 비슷한 구조를 갖고 있는 모드들의 원본 파일들을 이 시스템이 읽고 정규화하여 내부 Iris 표준 파일로 만든 뒤 그걸 QG,DVF가 소비하도록 만드는 것임

외부 모드 데이터는 JSON,SQLite 등 열린 형식을 허용하고 그걸 내부에서 .Iris 같은 내부 포맷으로 정규화하는 방식으로 진행
또한 출력이 필요할 경우 또 다시 Json, SQLite 방식으로 출력하도록 강제할 예정

6. Frame - Pulse의 기능을 사용하여 Lua 대형 모드팩을 관리하기 쉽게 만들수있도록 지원해주는 모드

Frame은 PZ 모드팩의 상태(State)를 시간(Time) 축 위에서 기록/비교/되돌릴 수 있게 만드는 레이어.

“모드 자체”가 아니라 모드팩(환경)의 구성을 1급 객체로 다룸.

절대 하면 안 되는 것: 원인 지목, 추천/정답 제시, 자동 해결(심판/정책 도구로 변질 금지).

Frame이 다루는 건 환경(팩 상태):

모드 목록/순서/출처(창작마당/로컬 등)

팩 제작자가 포함한 설정 파일들(ini 등)

월드(세이브)는 별개로 두고 Frame에서 커버하지 않기로 합의.

팩의 기본 설정은 원본으로 보존.

유저가 바꾸고 싶으면 원본을 직접 수정하는 게 아니라:

복사본(내 설정 파일)을 만들어서 수정.

문제 생기면 내 설정 파일만 삭제/비활성화 → 즉시 기본값 복귀.

값 수정은 Frame이 “설정 에디터”가 되지 않게:

메모장/워드패드 등 외부 편집기로 열기 중심(리소스팩 폴더 열기 같은 느낌).

안내 문구는 “기본 설정은 보호, 내 설정은 언제든 되돌릴 수 있음”으로 간결하게.

창작마당 모드 파일을 Frame이 원본 그대로 저장/배포하는 완전 재현성은 피하기로 합의(권리/약관/삭제 모드 문제).

Frame이 하는 재현성은:

1. 재구성: 목록/순서/설정으로 다시 맞추기

2. 같은지 확인: “그때와 지금이 같은 상태인지”를 확인할 수 있는 표식(지문) 저장

옛 버전(예: 1.5.0) 고정 문제는 Frame이 자동으로 해결하지 않고,

“그때와 다름”을 확실히 드러내는 쪽으로 정리.

Frame은 “정상/비정상, 원인/범인, 권장/최적, 해결/진단” 같은 판단/추천 톤을 금지.

대신 “기준점(직접 저장), 자동 저장(주기), 달라짐/비교, 되돌리기/계속” 같은 사실+행동 언어만 사용.

작동 방식

메인 - 수동 스냅샷 + 세션 기반

보조 - 자동 스냅샷(보험용) + 이벤트 기반(태그용)

자동은 주기(5/10/30/60분)마다 자동 스냅샷 생성.

최근 10개만 보관(롤링 보관)으로 노이즈/용량 통제.

자동 스냅샷은 품질이 낮은 게 아니라:

직접 저장(의도적 기준점) vs 자동 저장(주기 기록)
→ 의도/역할 차이만 있고 품질 차이는 없음.

UI에서 자동은 기본 숨김, “자동 저장도 표시하기” 같은 옵션으로 볼 수 있게 분리.

전용 .frame 확장자를 생태계 표준으로 밀지 않기로 했고,

공유 파일은 표준 ZIP으로:

manifest.json (요약/메타/모드 목록/순서/출처/직접·자동 구분/오버라이드 포함 여부)

baseline/ (팩 기본 설정)

overrides/ (내 설정 포함 시)

fingerprint.json (같은지 확인용 표식)

자동 스냅샷 공유는 기본 플로우가 아니라:

옵션으로만 지원(체크박스나 별도 버튼).

유저 오버라이드 공유도 허용:

기본 허용하되, 포함 여부를 명시하고 혼동 없게 분리.

공유는 ZIP+JSON, 가져오기 시 검증 후 내부적으로 .frame 캐시로 변환(안전/일관성↑)

8. Canvas - 리소스팩 관련 기능을 담당할 별개의 모듈

좋다. 현재까지 논의한 Canvas 설계를, 이번 세션 합의까지 반영해 다시 정리하면 아래처럼 된다.

1. Canvas의 위치

Canvas는 Pulse 생태계의 별도 하위 모듈이다.

Pulse는 하위 모듈을 참조하거나 의존하지 않는 얇은 기반 플랫폼이어야 하고, Canvas는 Pulse capability만 소비하는 spoke로 남는다. Hub & Spoke, SPI, 역할 침범 금지, 호환성 우선 원칙을 그대로 따른다.

여기서 중요한 현재 기준은 다음이다.

리소스팩 축을 하기로 하면 처음부터 Canvas로 시작
하지 않기로 하면 Pulse에서도 제품 축으로서의 리소스팩 기능은 제거

즉, Cortex가 초기 리소스팩 기능을 임시 수용했다가 Canvas로 이관하는 경로는 사용하지 않는다.

2. Canvas의 정체성

Canvas는 리소스 제작 툴이 아니다.
그리고 단순 리소스팩 로더도 아니다.

현재 기준의 정체성은 이쪽이다.

Canvas = 리소스팩을 사용자-facing 1급 객체로 다루되, 그 리소스팩이 실제로 만든 최종 적용 상태를 검증·비교·설명하는 리소스 적용 상태 관리 플랫폼/관리 레이어

여기서 층위는 다음처럼 분리한다.

구현 최소 단위: AssetEntry
실제 리소스 파일/엔트리/대상 경로/식별 단위
사용자-facing 핵심 객체: ResourcePack
사용자가 만들고, 불러오고, 공유하고, 비교하는 단위
철학적 최상위 개념: ResourceState
여러 리소스팩이 합쳐져 실제 게임에 최종 적용된 결과 상태

즉, 최상위 개념과 사용자에게 직접 보이는 1급 객체를 같은 것으로 두지 않는다.

한 줄로 줄이면:

Canvas는 ResourcePack을 통해 ResourceState를 관리하는 모듈이다.

3. Canvas가 하지 않는 것

이건 강하게 봉인한다.

Canvas는:

Photoshop/GIMP/Blender/TileZed 같은 제작 툴과 경쟁하지 않음
이미지/사운드/모델을 직접 만드는 에디터가 아님
자동 병합, 정답 추천, 최적 순서 제안 같은 정책/심판 도구가 아님
Frame처럼 스냅샷/재현 책임을 먹지 않음
Cortex처럼 잡다한 편의 기능의 격리 구역이 아님

즉 Canvas는 “만드는 툴”이 아니라 “만든 결과물이 안 깨지고 실제로 어떻게 적용되는지 보이게 하는 플랫폼” 이다.

4. Canvas가 대응해야 하는 핵심 문제 3개

Canvas가 겨냥하는 pain point는 3개다.
중요한 점은 이 3개를 단일 마스터 점수로 접지 않고, 독립 판정 축으로 둔다는 것이다.
Canvas는 종합 진단 도구이지만, 종합 진단은 설명 레이어에서 묶는 것이지 판정 자체를 하나로 압축하는 것은 아니다.

첫째: 최종 적용 결과 / 충돌 / 로드 순서 가시성 부족

사람들이 가장 자주 답답해하는 건 “지금 뭐가 최종적으로 먹히는지 모르겠다”는 문제다.

Canvas는 여기에 대해:

리소스 인덱싱
최종 적용 상태 계산
충돌 분석
팩 영향 범위 표시
가상 비교
설명 레이어

를 가진다.

핵심은 누가 최종 승자인지 계산하고, 왜 그렇게 됐는지 보여주는 것이다.

둘째: 패킹 / 경로 / 구조 / ID가 예민해서 제작 과정이 쉽게 깨짐

이 문제는 제작 툴로 푸는 게 아니라, 제작 안전 레이어로 푼다.

Canvas는 여기에 대해:

구조 검증
경로 검증
패킹 검증
ID/중복 충돌 탐지
프리플라이트 빌드
설명형 진단 리포트

를 가진다.

즉 “왜 안 먹는지, 어디가 위험한지, 지금 상태로 배포 가능한지”를 게임에 넣기 전에 드러내는 역할이다.

셋째: 버전 / 서버 / 배포 불일치

이 문제는 “같은 팩인 줄 알았는데 실제론 다른 상태”를 비교해 드러내는 방향으로 정리한다.

Canvas는 여기에 대해:

배포 manifest 생성
로컬 작업본 ↔ 빌드 산출물 비교
서버 ↔ 클라이언트 상태 비교
불일치 유형 분류
설명형 차이 리포트

를 가진다.

즉 고쳐주는 모듈이 아니라, 비교해서 차이를 설명하는 모듈이다.

판정 구조

Canvas는 최소한 아래의 판정 축을 독립적으로 가진다.

적용 상태 판정
지금 실제 게임에서 무엇이 최종 적용되었는가
제작 안전 판정
이 산출물/작업본의 구조와 패킹이 유효한가
배포 일치 판정
기대 상태와 실제 상태, 또는 A/B 상태가 서로 같은가

종합 리포트는 이 셋을 묶어 설명할 수 있지만,
그 자체를 하나의 숨겨진 최종 점수나 단일 합격/불합격으로 덮지 않는다.

5. Pulse와 Canvas의 경계

Pulse는 기반 기능만 제공하고, 결정/정책을 하지 않는다.

현재 기준으로 보면:

Pulse가 담당
활성 리소스팩/순서/출처 조회
리소스 식별자 정규화
해시/fingerprint 유틸
리소스 변경/리로드 이벤트
Networking 기반 상태 교환
SPI 인터페이스
공통 진단 출력 구조

즉 읽고, 식별하고, 통지하고, 연결하는 기반

Canvas가 담당
인덱싱
정규화
최종 상태 계산
충돌 분석
프리플라이트 검증
비교
설명
리포트 UX

즉 비교하고, 계산하고, 보여주고, 설명하는 상위 레이어

6. 구현 구조

Canvas는 Iris처럼 100% Lua 제약을 갖지 않는다.
따라서 Java와 Lua를 혼용한다.

권장 구조는 다음이다.

Java core
외부 산출물 import
내부 정규화
.cvb 생성/로드
인덱싱
최종 상태 계산
충돌 분석
경로/구조/ID/패킹 검증
비교 엔진
fingerprint 계산
진단 결과 생성

즉 머리와 심장

Lua shell
인게임 패널
비교/진단/검증 결과 UI
브라우징/탐색 진입점
설명형 리포트 렌더링
사용자의 인게임 조작면

즉 손과 얼굴

중요한 원칙은, 진실의 소스와 계산 로직은 Java core가 소유하고, Lua는 그 결과를 소비하는 얇은 표면으로 둔다는 점이다.

7. 입력 / 내부 / 출력 / 공유 포맷

여기서 중요한 건 외부 포맷과 내부 포맷, 그리고 기대 상태와 실제 상태를 구분하는 것이다.

입력

Canvas는 둘 다 읽는다.

A. 배포 산출물 / 외부 입력
프로젝트 폴더
외부 파일 구조
.pack
ZIP + manifest/json

이건 의도된 상태 / 기대 상태 / 배포 대상을 읽는 입력이다.

B. 런타임 활성 상태
현재 활성 리소스팩
현재 로드 순서
현재 출처
현재 게임이 실제로 보고 있는 적용 상태

이건 실제 상태 / 현재 현실을 읽는 입력이다.

즉 Canvas는
배포 산출물 import도 보고, 런타임 활성 상태도 본다.
둘은 상호배타적인 관계가 아니며, 각각 독립적으로 판정하고 서로 비교한다.

내부

Canvas는 필요하면 내부적으로 정규화된 상태를 만든다.

내부 모델
캐시
비교 상태
진단 상태
선택적 .cvb 내부 포맷

여기서 .cvb는 외부 표준이 아니라 내부 캐시/세션/분석 번들이다.

.cvb의 역할은:

외부 입력과 런타임 상태를 정규화한 내부 번들
비교/진단/재현용 내부 캐시
문제 공유용 보조 포맷

외부 공개 표준로 밀지 않는다.

출력

최종 출력은 가능하면 PZ 네이티브 포맷인 .pack 중심으로 간다.
그리고 옆에 manifest.json을 둔다.

즉:

출력: .pack + manifest.json
필요하면 ZIP으로 감쌈
공유

공유 기본값은 Canvas 전용 포맷이 아니다.

기본 공유:

ZIP + .pack + manifest.json

선택적 공유:

소스 ZIP + manifest
문제 재현/분석용 .cvb 번들

즉 .cvb는 Frame의 .frame과 비슷하게 입출력 표준이 아니라 내부 처리/문제 공유용 보조 포맷으로만 제한한다.

8. .cvb의 기본 원칙

.cvb는 다음처럼 본다.

정체성: Canvas 내부 정규화 번들
컨테이너: ZIP + JSON 묶음
신뢰 모델:
외부 입력 → 강한 비신뢰 검증
검증 통과 후 메모리 내부 모델 → 신뢰 상태
.cvb 재로드 → 내부 캐시 재검증
버전 구조:
format_version
schema_version
producer_version

즉 .cvb는 “절대 신뢰 파일”도 아니고 “외부 공개 표준”도 아니다.
내부 기원 캐시 파일이지만, 재로드 시 최소 검증이 필요한 내부 분석 번들이다.

9. Canvas의 현재 설계 철학을 한 문장으로 요약하면

Canvas는 외부 툴이 만든 리소스팩 산출물과 현재 런타임 활성 상태를 함께 읽어, 경로/구조/ID/패킹/배포 문제를 검증하고, 최종 적용 상태와 충돌을 계산·비교·설명하는 리소스 적용 상태 관리 플랫폼이다.

10. 현재 시점에서 남은 것

개념 설계는 거의 끝났고, 남은 건 구현 직전 압축이다.

남은 핵심은:

ResourceState / ResourcePack / AssetEntry를 코드/문서 양쪽에서 canonical object model로 봉인
독립 판정 3축의 출력 계약 고정
.cvb의 최소 내부 파일 트리와 meta.json 최소 필드 확정
Pulse/Canvas capability 경계 문장 봉인
manifest 최소 필드 확정

여기까지가 지금 기준의 Canvas 설계다.

Fuse, Echo는 Pulse에 의존하는 Java를 사용하는 모드기에 Pulse의 경량형 라이브러리가 탑재되어야함.

7. Pulse, Echo를 프좀갤에 공개.

9. 이후 Frame과 Cortex를 프좀갤에 Github 링크로 선공개.
