Pulse 생태계 설계 철학

이는 절대 어기면 안됨

[1] Echo, Fuse, Nerve+, Iris, Frame, Cortex, Canvas는 Pulse의 기능들을 참조하고 의존성을 가져도 되지만 Pulse는 Echo, Fuse, Nerve+, Iris, Frame, Cortex, Canvas를 참조하거나 의존성을 가지면 절대 안됨.

[2] 기본적으로 Hub & Spoke 구조를 지켜야함. Echo, Fuse, Nerve+, Iris, Frame, Cortex, Canvas는 절대 서로를 참조하거나 의존성을 가지면 안됨. 어떠한 기능을 구현하기 위해 참조를 해야할경우 중간에 Pulse를 경유하는 방식으로 구현해야함.

[3] SPI 구조를 사용해야함.

[4] 타 모드와의 호환성이 1순위여야함. 

[5] Pulse 생태계의 각 모듈은 각자의 역할을 고수하고 Pulse 생태계 내부의 타 모듈들의 역할을 침범하면 안됨.(구조 경계를 선명화하기 위한 외부 모드에 적용되지 않는 내부 규정임)

각 모듈별 세부 규정

1. Pulse - 모드로더 겸 플랫폼의 역할 [기반적 기능]

Pulse의 유일한 정체성은 "자유도". 완전한 샌드박스형 모딩 플랫폼이 목표임

아무 목표도 없는, 다만 무너지지 않고 탄탄한 기반만 제공하는 '휑한 들판'같은 플랫폼

오로지 기반적 기능만 가져야하고 이를 API 등의 형태로 타 모드에 제공해야함

모드적 기능은 포함하면 절대 안됨

-> 버전 업데이트에 강한 내성을 가지게 됨

추가할 만한 편의성 기능들은 모두 Cortex로 이전.

목표는 Fabric형 얇은 플랫폼

결정/정책은 절대 하지 않음.

퍼스트 파티 모드라 잘 돌아가는게 아닌 플랫폼이 좋아서 퍼스트 파티 모드가 잘 돌아가도록 만들어야함.

목표는 기존의 모딩 생태계(Lua 기반 모드)와 새로운 모딩 생태계(Mixin 기반 모드)를 융합하여 모더들에겐 더 큰 모딩 자유도를, 일반 유저들에겐 더 큰 모드 선택지를 제공하는것.

나중에 리소스팩 관련으로 지원 기능을 추가할 예정

JVM+Lua 혼용 금지

5. Iris - [100% Lua를 사용하는 위키피디아형 모드]

"질문은 다르되, 대답의 태도는 같다"

기본 바닐라 툴팁 위에 Alt 키를 눌렀을 때만 Iris 툴팁이 표시되어 용도와 더 자세한 설명이 나오도록 해야함. 길이는 최대 4줄.

아이템을 오른 클릭을 했을때 "Iris 메뉴에서 더보기" 버튼을 클릭했을때 Iris 모드 메뉴(위키)가 나타나 더욱 세부적인 정보를 제공(실용적인 정보만)

Iris 메뉴는 대분류 - 소분류 - 아이템 목록 - 아이템 설명 순으로 구성될 예정이고 각 칸의 크기는 아이템 설명 > 대분류 > 소분류 = 아이템 목록 순으로 이루어질 예정

위키피디아 모드인 이상 게임플레이에 실질적인 도움이 되는 정보를 담아야 함.

대신 해석, 권장, 비교는 엄격히 금함.

Iris 메뉴에 담기는 정보들은 전부 툴팁보다는 상세해야하며 툴팁의 확장선이라는 느낌보단 위키라는 느낌의 정보가 들어있어야함.

Iris 시스템은 총 3단계로 구성됨

그리고 그 각 단계 또한 여러개의 층위로 구성됨

1단계 증거 시스템은 2개의 동일 2트랙으로 구성됨

1-1. 레시피 기반 증거 시스템 1-2. 우클릭 행동 기반 증거 시스템

이 두 계층은 서로 독립적이며, 둘 사이엔 우열이 없는 증거 시스템 하위의 동급 시스템이어야 함.

2단계 아이템 분류 시스템

2-1. 대분류 2-2. 소분류 2-3. 아이템 목록 2-4. 아이템 설명란

3단계 아이템 설명란

3-1. 아이템의 기초 설명 3-2. 주 소분류 설명 3-3. 아이템의 개별 설명 3-4. 아이템의 관련 레시피나 관련 우클릭 행동 등의 아이템의 상호작용 3-5. 아이템의 내부 정보 (PZ 내부 분류 정보, Iris 내부 분류 정보 등)

아이템 설명란의 각 계층은 선행 계층의 필터 결과물이 아니라, 각자 고유한 정보 유형을 담당하는 독립 층임. 다만 실용성 기준상 1계층과 2계층만으로 용도 설명이 충분한 경우 3계층과 4계층은 침묵할 수 있음. 5계층은 내부 정보층이므로 항상 존재함.
단, 고유 정보 유형이라고 해서 굳이 타 계층을 배제하여 서술할 필요는 없음.

추후 모드 시장 확장 시스템 역시 구현할 예정

전략은 바닐라와 비슷한 구조를 갖고 있는 모드들의 원본 파일들을 이 시스템이 읽고 정규화하여 내부 Iris 표준 파일로 만든 뒤 그걸 QG,DVF가 소비하도록 만드는 것임

외부 모드 데이터는 JSON,SQLite 등 열린 형식을 허용하고 그걸 내부에서 .Iris 같은 내부 포맷으로 정규화하는 방식으로 진행
또한 출력이 필요할 경우 또 다시 Json, SQLite 방식으로 출력하도록 강제할 예정

6. Frame - 대형 모드팩 관리용 모듈

Frame은 PZ 모드팩의 상태를 시간축 위에서 기록·비교·되돌릴 수 있게 만드는 모드팩 상태 관리 모듈이다.

Frame은 다음과 같은 특정 시점의 모드팩 환경 전체를 1급 객체로 다룸.
- 모드 목록
- 모드 순서
- 모드 출처
- 팩 제작자가 포함한 설정 파일
- 사용자가 만든 설정 오버라이드
- 그 상태가 이전 기준점과 같은지 확인하기 위한 fingerprint

그 중 핵심 객체는 PackState다.

PackState엔 다음이 포함된다.
- 어떤 모드가 있었는가
- 어떤 순서였는가
- 어디에서 온 모드였는가
- 어떤 설정 상태였는가
- 현재 상태가 그때와 같은가

Frame의 목적은 언제 어떤 상태였고, 지금 무엇이 달라졌는지 드러내는 것이다.

하는 일

Frame은 다음 역할을 담당한다.

1. 상태 기록
- 사용자가 직접 저장한 기준점 기록
- 세션 또는 주기 기반 자동 저장
- 모드 목록 / 순서 / 출처 / 설정 상태 보존
- fingerprint 기반 동일성 확인 표식 생성

2.상태 비교
- 이전 기준점과 현재 상태 비교
- 모드 목록 변화 확인
- 순서 변화 확인
- 설정 변화 확인
- 같은 상태인지 다른 상태인지 표시

3.상태 복원
-저장된 기준점으로 모드팩 구성을 되돌릴 수 있게 지원
- 기본 설정과 사용자 오버라이드를 분리해 복구 가능성 확보
-삭제나 변경으로 인해 완전 복원이 불가능한 경우, 그 차이를 명확히 드러냄

4.공유
- 모드팩 상태를 열린 포맷으로 내보냄
- 가져오기 시 검증 후 내부 상태로 정규화
- 포함된 설정과 사용자 오버라이드의 범위를 명확히 표시

하지 않는 일

Frame은 다음 역할을 맡지 않는다.

- 어떤 모드가 문제의 원인인지 지목하지 않는다.
- 정상 / 비정상 판정을 하지 않는다.
- 추천 설정, 최적 순서, 정답 구성을 제시하지 않는다.
- 자동 해결 도구가 되지 않는다.
- 모드 파일 원본을 그대로 저장·배포해 완전 재현성을 보장하려 하지 않는다.
- 월드나 세이브 상태를 관리하지 않는다.
- 설정 에디터가 되지 않는다.
- 런처, 설치기, devkit, 진단기 역할로 확장하지 않는다.
- Fuse / Nerve 같은 런타임 안정화 역할을 흡수하지 않는다.
- Canvas처럼 리소스팩 최종 적용 상태를 계산하지 않는다.

Frame은 심판 도구가 아니라 기록과 복원 도구다.

설정 관리 원칙

Frame은 팩의 기본 설정을 원본으로 보존한다.

사용자가 설정을 바꾸고 싶을 경우, 원본을 직접 수정하지 않고 사용자 오버라이드 파일을 만들어 수정하는 방향을 기본으로 한다.

문제가 생기면 사용자 오버라이드만 삭제하거나 비활성화해 기본값으로 돌아갈 수 있어야 한다.

Frame은 설정값을 직접 편집하는 UI를 중심으로 두지 않는다.
필요한 경우 외부 편집기로 열 수 있게 하고, Frame은 기본 설정과 사용자 설정의 경계, 변경 여부, 복구 가능성을 관리한다.

재현성 원칙

Frame은 창작마당 모드 파일을 원본 그대로 저장·배포하는 방식의 완전 재현성을 목표로 하지 않는다.

Frame이 제공하는 재현성은 두 가지다.

1. 재구성
- 모드 목록, 순서, 출처, 설정을 기준으로 그때의 환경에 가깝게 다시 맞춘다.

2.동일성 확인
- fingerprint를 통해 지금 상태가 저장된 기준점과 같은지 확인한다.

모드가 삭제되었거나, 창작마당의 버전이 바뀌었거나, 과거 버전을 더 이상 받을 수 없는 경우 Frame은 이를 자동으로 해결하지 않는다.
대신 “그때와 지금이 다르다”는 사실을 명확히 드러낸다.

스냅샷 원칙

Frame의 기록은 크게 두 종류로 나뉜다.

1. 수동 스냅샷
- 사용자가 직접 저장한 의도적 기준점
- Frame에서 가장 중요한 공식 기록

2. 자동 스냅샷
- 세션 또는 주기 기반으로 남기는 안전망
- 복구와 변화 추적을 돕는 보조 기록

자동 스냅샷은 수동 스냅샷보다 품질이 낮은 기록이 아니다.
다만 의도와 역할이 다르다.

수동 스냅샷은 사용자가 선언한 기준점이고, 자동 스냅샷은 예상치 못한 문제에 대비하는 시간축 안전망이다.

UI에서는 이 둘을 혼동하지 않도록 분리해 보여준다.

언어 원칙

Frame은 다음과 같은 판단 언어를 사용하지 않는다.

- 정상 / 비정상
- 원인 / 범인
- 권장 / 최적
- 해결 / 진단
- 문제 모드
- 자동 수정

Frame은 대신 사실과 행동 중심의 언어를 사용한다.

- 기준점
- 자동 저장
- 달라짐
- 비교
- 되돌리기
- 계속
- 포함됨
- 빠짐
- 순서 변경
- 설정 변경

즉 Frame의 언어는 처방이 아니라 기록에 가깝다.

Pulse와의 관계

Pulse가 Frame을 위해 제공할 수 있는 영역은 다음과 같다.

- 활성 모드 목록 조회
- 모드 순서 조회
- 모드 출처 조회
- 설정 파일 위치 조회
- 파일 해시 / fingerprint 유틸
- 파일 변경 이벤트
- 저장 / 불러오기 관련 기반 capability
- SPI 인터페이스
- 공통 진단 출력 구조

이에 따라 Frame이 담당하는 영역은 다음과 같다.

- PackState 모델링
- 스냅샷 생성
- 상태 비교
- 설정 오버라이드 관리
- fingerprint 생성과 비교
- 공유 패키지 생성
- 가져오기 검증
- 사용자-facing 복원 UX

입력 / 내부 / 출력

Frame은 외부 공유 파일과 현재 로컬 모드팩 상태를 모두 읽는다.

입력

Frame의 입력은 크게 두 종류다.

첫째, 현재 로컬 상태다.

- 현재 활성 모드 목록
- 현재 모드 순서
- 현재 모드 출처
- 현재 설정 파일
- 현재 사용자 오버라이드
- 현재 fingerprint

둘째, 외부 공유 상태다.

- ZIP + manifest
- baseline 설정
- overrides 설정
- fingerprint
- 메타데이터

현재 로컬 상태는 실제 상태이고, 외부 공유 상태는 기대 상태 또는 가져올 기준점이다.
Frame은 이 둘을 비교하고, 같음과 다름을 표시할 수 있다.

내부

Frame은 필요하면 외부 공유 파일과 현재 상태를 내부 모델로 정규화한다.

내부 상태는 다음을 포함할 수 있다.

- 내부 PackState 모델
- 스냅샷 캐시
- 비교 상태
- fingerprint 상태
- 선택적 .frame 내부 캐시

.frame은 외부 공개 표준이 아니라 Frame 내부 정규화 캐시다.
외부 공유 파일을 가져온 뒤 검증과 일관성을 위해 내부적으로 사용할 수 있지만, 생태계 표준 포맷으로 강제하지 않는다.

출력 / 공유

Frame의 기본 공유 포맷은 열린 형식을 우선한다.

기본 공유는 다음을 사용한다.

- ZIP
- manifest.json
- 기준 설정 파일
- 선택적 사용자 오버라이드
- fingerprint.json

사용자 오버라이드 공유는 허용할 수 있지만, 포함 여부를 명확히 표시한다.
자동 스냅샷 공유는 기본 플로우가 아니라 선택 옵션으로 둔다.

즉 Frame의 외부 공유는 ZIP + JSON 중심이고, .frame은 내부 처리와 검증을 위한 보조 포맷으로 제한한다.

8. Canvas - 리소스팩 관련 기능을 담당할 별개의 모듈

Canvas는 ResourcePack을 통해 ResourceState를 관리하는 리소스 적용 상태 관리 모듈이다.

여기서 핵심 개념은 다음처럼 분리한다.

AssetEntry: 실제 리소스 파일, 엔트리, 대상 경로, 식별 단위
ResourcePack: 사용자가 만들고, 불러오고, 공유하고, 비교하는 사용자-facing 핵심 객체
ResourceState: 여러 리소스팩이 합쳐져 실제 게임에 최종 적용된 결과 상태

즉 Canvas의 사용자-facing 1급 객체는 ResourcePack이고, 철학적 최상위 개념은 ResourceState다. Canvas는 ResourcePack을 직접 다루게 하되, 궁극적으로는 실제 적용 결과인 ResourceState를 계산·비교·설명한다.

Canvas의 관장하는 핵심 축은 다음과 같다:

1.적용 상태 판정
- 현재 활성 리소스팩 조회
- 로드 순서와 출처 반영
- 최종 적용 상태 계산
- 충돌 분석
- 어떤 리소스가 최종 승자인지 설명

2. 제작 안전 판정
- 경로 검증
- 구조 검증
- ID / 중복 충돌 탐지
- 패킹 검증
- 배포 전 프리플라이트 진단

3. 배포 일치 판정
- 로컬 작업본과 빌드 산출물 비교
- 서버와 클라이언트 상태 비교
- 기대 상태와 실제 상태 비교
- manifest / fingerprint 기반 차이 설명

이 세 판정 축은 독립적으로 유지한다. Canvas는 종합 리포트에서 이들을 묶어 설명할 수는 있지만, 하나의 숨겨진 마스터 점수나 단일 합격/불합격으로 압축하지 않는다.

하지 않는 일

Canvas는 다음 역할을 맡지 않는다.

- 이미지, 사운드, 모델을 직접 만드는 에디터가 되지 않는다.
- 단순 리소스팩 로더로 축소되지 않는다.
- 자동 병합, 정답 추천, 최적 로드 순서 제안 같은 정책/심판 도구가 아니다.
- Frame처럼 모드팩 스냅샷이나 재현 책임을 맡지 않는다.

Canvas는 “리소스를 만드는 도구”가 아니라, 만든 결과물이 실제로 어떻게 적용되고, 어디서 깨지며, 무엇과 충돌하는지 보이게 하는 관리 레이어다.

Pulse와의 관계

Pulse는 Canvas를 위해 다음과 같은 기반 capability만 제공한다.

- 활성 리소스팩 / 순서 / 출처 조회
- 리소스 식별자 정규화
- 해시 / fingerprint 유틸
- 리소스 변경 / 리로드 이벤트
- Networking 기반 상태 교환
- SPI 인터페이스
- 공통 진단 출력 구조

이에 따라 Canvas가 담당하는 영역은 다음과 같다.

- 리소스 인덱싱
- 내부 정규화
- 최종 적용 상태 계산
- 충돌 분석
- 프리플라이트 검증
- 상태 비교
- 설명형 리포트
- 사용자-facing UX

즉 Canvas는 비교하고, 계산하고, 보여주고, 설명하는 상위 레이어다.

입력 / 내부 / 출력

Canvas는 외부 산출물과 런타임 활성 상태를 모두 읽는다.

입력

Canvas의 입력은 크게 두 종류다.

첫째, 배포 산출물 / 외부 입력이다.

- 프로젝트 폴더
- 외부 파일 구조
- .pack
- ZIP + manifest / JSON

이는 의도된 상태, 기대 상태, 배포 대상을 읽는 입력이다.

둘째, 런타임 활성 상태다.

- 현재 활성 리소스팩
- 현재 로드 순서
- 현재 출처
- 현재 게임이 실제로 보고 있는 적용 상태

이는 실제 상태, 현재 현실을 읽는 입력이다.

Canvas는 이 둘을 상호배타적으로 보지 않는다. 각각 독립적으로 판정하고, 필요하면 서로 비교한다.

내부

Canvas는 필요하면 외부 입력과 런타임 상태를 다음과 같은 내부 모델로 정규화한다.

- 내부 ResourceState 모델
- 캐시
- 비교 상태
- 진단 상태
- 선택적 .cvb 내부 번들

.cvb는 외부 공개 표준이 아니라 Canvas 내부 정규화 번들이다.
외부 입력과 런타임 상태를 정규화한 내부 캐시, 비교/진단/재현용 분석 번들, 문제 공유용 보조 포맷으로만 사용한다.

.cvb는 절대 신뢰 파일도 아니고 외부 표준도 아니다. 내부 기원 캐시 파일이더라도 재로드 시 최소 검증을 거친다.

출력 / 공유

Canvas의 기본 출력은 가능하면 PZ 네이티브 포맷을 존중한다.

기본 출력은 다음을 우선한다.

- .pack
- manifest.json
- 필요 시 ZIP 래핑

기본 공유는 다음을 사용한다.

- ZIP + .pack + manifest.json

선택적 공유는 다음을 허용할 수 있다.

- 소스 ZIP + manifest
- 문제 재현 / 분석용 .cvb 번들

즉 .cvb는 Frame의 .frame과 유사하게 외부 입출력 표준이 아니라, 내부 처리와 문제 공유를 위한 보조 포맷으로 제한한다.

JVM+Lua 혼용 허용, 그러나 최종 적용 상태 계산, 충돌 분석, 검증, 비교 같은 판정 로직은 Java Core가 소유하고, Lua는 그 결과를 표시·탐색하는 사용자-facing 표면으로 둔다.
