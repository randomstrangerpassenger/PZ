# DVF / Registry Handoff Contract

DVF owns contract-valid facts, decisions, profile rules, body-plan resolution, and the rendered 3-3 body. The compiler trace is `rendered.meta`; no separate compiler receipt exists.

Before Registry adoption, every DVF output is a candidate. Registry observes the candidate through a read-only observation receipt and owns only role classification, identity verification, and candidate-to-current promotion conditions. The observation receipt is not a compiler receipt, Registry seal, or authority source.

This closure has no legal real current-writer callsite. `compose_layer3_text.py` rejects every resolved real current protected target before creating a directory or writing bytes, even when a receipt is supplied. Its receipt option is restricted to a one-use `fixture_only` transaction below the registered attempt's `phase4/wp5` root. Operational current writes require a separate reviewed serialized-cutover design.

This contract makes no Runtime Compatibility, Publish Boundary, package publication, release-readiness, or public-text claim.
